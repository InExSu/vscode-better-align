# TS Structural Alignment Engine

## Детерминированное идемпотентное структурное выравнивание кода

Реализуй structural alignment engine на TypeScript.

Система должна быть:

* детерминированной
* идемпотентной с нормализацией пробелов: `whitespace-normalized`
* структурно-осознанной
* чисто функциональной
* основанной на конечных автоматах (FSM)
* стабильной при многократном выполнении

Использовать:

* чистые функции
* явное неизменяемое состояние
* иерархический конечный автомат
* легковесное структурное сканирование по якорям-признакам (без токенизатора)

Запрещено:

* рекурсивные проходы форматирования
* парсинг только на основе регулярных выражений
* скрытое изменяемое состояние
* полноценный AST-парсер
* эвристики на основе пробелов
* выравнивание на основе предыдущего форматирования

Вывод:
ТОЛЬКО ПОЛНОСТЬЮ РАБОЧИЙ КОД.
БЕЗ ОБЪЯСНЕНИЙ.

══════════════════════════════════════════════════════════════════

ОБЩАЯ ИДЕЯ

Система должна выравнивать код:

* локально
* структурно
* детерминированно
* идемпотентно (с нормализацией пробелов)

Главное требование:

```ts
normalizeWhitespace(format(format(code))) === normalizeWhitespace(format(code))
```

Повторный запуск форматтера никогда не должен:

* добавлять пробелы сверх нормы
* сдвигать колонки после первого прохода
* расширять выравнивание
* менять хвостовые сегменты

Выравнивание выполняется НЕ глобально.

Код разбивается на локальные блоки.

Каждый блок выравнивается независимо.

══════════════════════════════════════════════════════════════════

ОБЪЯСНЕНИЕ ДЛЯ ЗАКАЗЧИКА

Форматтер должен делать код визуально читаемым,
но не ломать структуру.

Выравнивание должно происходить только там,
где строки действительно похожи по якорям-признакам.

Например:

ВХОДНЫЕ ДАННЫЕ:

```ts
let a = 1
let longName = 2
```

РЕЗУЛЬТАТ:

```ts
let a        = 1
let longName = 2
```

Но вот это:

```ts
let x = 1
return { a: 1 }
```

НЕ должно пытаться выравниваться вместе,
потому что это разные конструкции.

──────────────────────────────────────────────────────────────────
ВАЖНОЕ ПРАВИЛО
──────────────────────────────────────────────────────────────────

Совместимость строк определяется НЕ полным совпадением,
а общим левым префиксом якорей.

Например:

```ts
let z = 1
let pq = { start: 0, end: 0 }
```

Первая строка имеет:

```ts
['=']
```

Вторая:

```ts
['=', ':', ':']
```

Обе строки совместимы,
потому что общий префикс:

```ts
['=']
```

Результат:

```ts
let z  = 1
let pq = { start: 0, end: 0 }
```

══════════════════════════════════════════════════════════════════

ОБЪЯСНЕНИЕ ДЛЯ МАТЕМАТИКА

Пусть:

```ts
P = алфавит якорей выравнивания
```

Например:

```ts
P = {
  '=',
  ':',
  '=>',
  '==',
  '!=',
  ...
}
```

Для строки `f`:

```ts
V(f) = упорядоченный вектор якорей
```

Поиск якорей:

* слева направо
* жадный
* сначала самый длинный
* непересекающийся

Пример:

```ts
a: b = c
```

```ts
V(f) = [':', '=']
```

──────────────────────────────────────────────────────────────────
СОВМЕСТИМОСТЬ ПО ПРЕФИКСУ

Определим:

```ts
commonPrefix(A, B)
```

как максимальный общий левый префикс.

Примеры:

```ts
['=']
['=', ':']
→ ['=']

[':', '=']
[':', '=', ',']
→ [':', '=']

['=']
[':']
→ []
```

──────────────────────────────────────────────────────────────────
ОПРЕДЕЛЕНИЕ БЛОКА — ИТЕРАТИВНОЕ СУЖЕНИЕ ПРЕФИКСА

Блок формируется алгоритмом:

```
1. Инициализировать blockPrefix = V(firstLine)
2. Для каждой следующей строки:
   blockPrefix = commonPrefix(blockPrefix, V(line))
   если blockPrefix пуст → разбить блок
3. Выравнивать только якоря из финального blockPrefix
```

Блок = максимальная непрерывная последовательность строк,
для которой финальный `blockPrefix` не пуст.

Префикс блока может только уменьшаться.
Никогда не увеличиваться.

──────────────────────────────────────────────────────────────────
ИДЕМПОТЕНТНОСТЬ

Требуется:

```ts
normalizeWhitespace(F(F(x))) = normalizeWhitespace(F(x))
```

Рендеринг должен зависеть только от:

* канонических сегментов
* измеренных ширин
* финального общего префикса

Рендеринг НЕ должен зависеть от:

* предыдущих пробелов
* предыдущего выравнивания
* визуальных колонок предыдущего прохода

Следовательно дрейф невозможен.

ОБЪЯСНЕНИЕ ДЛЯ TYPESCRIPT ПРОГРАММИСТА

Архитектура:

* неизменяемое состояние
* явный конечный автомат
* без скрытых мутаций
* без рекурсивного форматирования
* без AST
* легковесный структурный сканер по якорям-признакам

Использовать:

* неизменяемые массивы
* размеченные объединения
* маленькие чистые функции

Пример:

```ts

```

Поиск якорей:

* детерминированный
* структурно-осознанный (игнорировать внутри строк/комментариев)
* непересекающийся

СТРУКТУРНЫЕ ОБЛАСТИ ВИДИМОСТИ (SCOPES)

Запрещено использовать примитивную модель:

```ts
parenDepth === 0
```

Потому что выравнивание внутри:

* `()`
* `[]`
* `<>`

часто нужно и полезно.

Вместо запрета по глубине нужна:

* модель с учётом областей видимости
* контекстно-зависимое выравнивание
* управляемое политиками выравнивание

──────────────────────────────────────────────────────────────────
КЛАССИФИКАЦИЯ ОБЛАСТЕЙ ВИДИМОСТИ

Нужен классификатор структурных областей видимости.

Например:

```ts
type ScopeKind =
  | 'ArrayLiteral'       // Массивный литерал
  | 'CallArgs'           // Аргументы вызова
  | 'Expression'         // Выражение
  | 'FunctionArgs'       // Аргументы функции
  | 'GenericParams'      // Generic-параметры
  | 'ObjectLiteral'      // Объектный литерал
  | 'Root'               // Корень
  | 'Tuple'              // Кортеж
  | 'TypeLiteral'        // Типовой литерал
```

══════════════════════════════════════════════════════════════════

ПОЛИТИКА ВЫРАВНИВАНИЯ

Выравнивание разрешено НЕ по глубине,
а по политике текущей области видимости.

Например:

```ts
type ScopePolicy = {
  readonly allowAlignment: boolean
}
```

Вместо:

```ts
parenDepth === 0
```

должно быть:

```ts
scopePolicy[currentScope].allowAlignment === true
```

МАСКИРОВАНИЕ СТРОК / КОММЕНТАРИЕВ

Якоря внутри:

* строк
* шаблонных литералов
* комментариев

должны игнорироваться при поиске вектора якорей.

Нужна фаза маскирования по признакам:

* обнаружение начала/конца строкового литерала по кавычкам
* обнаружение `//` и `/* */` комментариев
* пропуск якорей внутри замаскированных регионов

Маскированные регионы:

* сохраняют длину для расчёта позиций
* заменяются символами-заполнителями при поиске якорей

Это гарантирует стабильные смещения.

──────────────────────────────────────────────────────────────────
ПРАВИЛА ФОРМИРОВАНИЯ БЛОКА

Пустая строка:

* сбросить текущий блок

Новая строка:

```
1. вычислить V(строка) — вектор якорей с учётом маскирования

2. вычислить:
   nextPrefix = commonPrefix(текущий.префикс, V(строка))

3. если nextPrefix пуст:
   - сбросить текущий блок
   - начать новый блок с этой строки

4. иначе:
   - добавить строку в текущий блок
   - префикс = nextPrefix
```

Префикс может только уменьшаться.

══════════════════════════════════════════════════════════════════

МОДЕЛЬ СЕГМЕНТОВ

Для каждого выравниваемого якоря j в строке:

```
[ключ_j][якорь_j][значение_j][разделитель_j][после_j]
```

──────────────────────────────────────────────────────────────────
ПОЛЯ

```
ключ_j:
- текст перед якорем
- trim()
- канонизированный

значение_j:
- текст после якоря
- до разделителя ИЛИ до следующего якоря
- trim()

разделитель_j:
- сам разделитель (запятая, точка с запятой и т.д.)
- сохраняется как есть

после_j:
- текст после разделителя
- trim()

хвост:
- всё после последнего выравниваемого якоря
- сохраняется ТОЧНО как в исходнике
```

══════════════════════════════════════════════════════════════════

ИЗМЕРЕНИЕ ШИРИНЫ

Для каждой колонки выравнивания в блоке:

```
i_Key_Length_Max[j] =
  максимальная visual-длина(ключ_j) по всем строкам блока

i_Value_Length_Max[j] =
  максимальная visual-длина(значение_j) по всем строкам блока
```

Visual-длина:

* учитывать ширину символов (emoji, CJK и т.д. = 2 единицы)
* обычные ASCII = 1 единица

Измеряется ТОЛЬКО по канонизированным сегментам.
НЕ по отрендеренным пробелам.

══════════════════════════════════════════════════════════════════

КАНОНИЧЕСКИЙ РЕНДЕРИНГ

Рендеринг ДОЛЖЕН быть каноническим.

Использовать:

```
padEndVisual(ключ_j, W_key[j])
+ якорь_j
+ ' '
+ padEndVisual(значение_j, W_value[j])
+ разделитель_j
+ после_j
```

──────────────────────────────────────────────────────────────────
ПРАВИЛА РЕНДЕРИНГА

* РОВНО ОДИН пробел после якоря
* Пробелы перед якорем определяются ТОЛЬКО `padEndVisual`
* Никогда НЕ сохранять старые пробелы из исходника
* Хвост строки (после последнего якоря) сохраняется как есть
* Пустые строки и строки только с пробелами сохраняются как есть

══════════════════════════════════════════════════════════════════

ОБЛАСТЬ ВЫРАВНИВАНИЯ

Выравниваются ТОЛЬКО якоря,
входящие в финальный общий префикс блока.

══════════════════════════════════════════════════════════════════

ОБРАБОТКА КРАЕВЫХ СЛУЧАЕВ

* Пустая входная строка → вернуть ""
* Строка только с пробелами/табами → сохранить как есть
* Несбалансированные скобки/кавычки → продолжить обработку с лучшим усилием, без падения
* Unicode / emoji в ключах или значениях → учитывать visual width при выравнивании
* Работа внутри VS Code: предполагается UTF-8, line endings \n

ПРИМЕР 3 — МНОГОКОЛОНОЧНОЕ ВЫРАВНИВАНИЕ

ВХОД:
```ts

```

РЕЗУЛЬТАТ:


══════════════════════════════════════════════════════════════════

ПРИМЕР 4 — ИТЕРАТИВНОЕ СУЖЕНИЕ ПРЕФИКСА

ВХОД:
```ts
a: b = c
xx: yy = zz
k: m
```

Векторы якорей:
```
Строка 1: [':', '=']
Строка 2: [':', '=']
Строка 3: [':']
```

Алгоритм:
1. blockPrefix = [':', '='] (после строки 1)
2. Строка 2: commonPrefix([':', '='], [':', '=']) → [':', '=']
3. Строка 3: commonPrefix([':', '='], [':']) → [':']
4. Финальный blockPrefix = [':']

РЕЗУЛЬТАТ:
```ts
a : b  = c
xx: yy = zz
k : m
```

Разбор:
- Колонка `:`: выровнены все три строки
- Колонка `=`: выровнены только строки 1 и 2 (у строки 3 нет этого якоря)
- Между `b` и `=` в строке 1 добавлены пробелы для выравнивания `=` под `=` из строки 2

ТРЕБУЕМЫЕ ТЕСТЫ

Должны включать тесты для:

* идемпотентности с нормализацией пробелов
* вложенных объектов
* generics
* комментариев (// и /* */)
* строк (", ', `)
* блоков из одной строки
* разбиения пустыми строками
* нескольких колонок выравнивания
* разделителей (запятые, точки с запятой)
* смешанных якорей
* итеративного сужения префикса
* частичной совместимости векторов
* расходящихся суффиксных якорей
* выравнивания с учётом областей видимости
* параметров функций
* типов-кортежей
* unicode/emoji в ключах (visual width)
* несбалансированных скобок (fallback)
* строк только с пробелами (сохранение)

Только TypeScript.

Без внешних зависимостей.

Строгая типизация обязательна.

* неизменяемые данные

* дрейфа пробелов
